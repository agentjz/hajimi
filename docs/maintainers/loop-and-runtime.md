# 主循环与运行时

## 主循环在干什么

主循环在 `src/agent/runTurn.ts`。

它负责做 5 件事：

1. 组装 system prompt
2. 读取运行时状态（任务板、队友、协议、后台、worktree、策略）
3. 调模型
4. 执行工具
5. 决定继续、暂停、yield、还是结束

## 这里最重要的几个约束

### 计划约束

变更型工具默认要求先有 `todo_write`，不能裸跑。

这不是提示词建议，而是工具执行前的机器拦截。

### inbox 注入

lead 和 teammate 的 inbox 都会在 turn 开始阶段统一注入。

worker 不再抢先消费 inbox，只做 `peek`。

### verification 状态机

文件改动和 mutating shell 会触发 verification 需求。

verification 不是无限循环，而是有预算、有暂停态的状态机。

### continuation / compression

如果工具步骤太多，turn 会自动续跑。

如果上下文太长，request context 会自动压缩。

### 交互等待反馈

交互模式下，用户发出输入后、模型真正开始输出前，会显示一个短暂的等待 spinner。

它只负责“等待阶段的反馈”，一旦 reasoning / assistant / tool / status 任何一种可见输出开始，就立即停止。

这个 spinner 属于 UI 层，不参与 agent 状态机本身。

## 为什么这些必须在同一层做

因为这些约束都属于“运行时调度规则”：

- 什么时候可以改
- 什么时候必须先计划
- 什么时候必须先处理 inbox
- 什么时候必须暂停给用户

这些不能散落到各个工具里，否则系统会越来越多例外。

## AI 维护者常见错误

### 错误 1

把状态逻辑直接写进某个工具里。

### 错误 2

靠 system prompt 提醒模型自觉，而不补机器约束。

### 错误 3

为了某个 case 临时加判断，却破坏主循环统一性。

## 改这层时怎么想

先问自己：

- 这是全局调度规则，还是单个工具细节？
- 这件事应该在 run loop 里统一处理吗？
- 这件事应该落到状态文件里吗？

如果答案偏向“全局”，优先改运行时，不要只改提示词。
