# S10 队友之间要有统一协议

## 原则

队友之间的协商不能靠自由文本临时约定，必须有统一的 request-response 模式。

## 为什么

如果每种协作都各写一套消息格式，系统会越来越多特例：

- 关闭队友是一套
- 计划审批是一套
- 以后资源申请、任务转交、冲突仲裁又是新的一套

最后问题不是“功能不够”，而是“协议不是唯一真相源”。

## 在本项目里

现在统一协议层由两部分组成：

- `src/team/requestStore.ts`：持久化的协议请求存储，统一维护 `pending -> approved/rejected` 状态机
- `src/team/messageBus.ts`：统一校验 `protocol_request` / `protocol_response` 消息格式与 `protocolVersion`

具体业务工具只是这个协议层的薄封装：

- `src/tools/shutdownRequestTool.ts`
- `src/tools/shutdownResponseTool.ts`
- `src/tools/planApprovalTool.ts`

## 当前规则

- 信息通知继续走 `send_message` / `broadcast`
- 任何会改变协作状态的沟通，都必须走协议化 request-response
- 所有协商都必须带 `request_id`
- 所有响应都必须引用原始 `request_id`
- 请求状态必须落盘，不能只存在消息里

## 结果

这样做以后，`shutdown` 和 `plan_approval` 不再是两套相似但分散的特例，而是同一个协议内核上的两个场景。
